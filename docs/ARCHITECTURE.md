**Key Design Principles:**

1. **Service Layer Pattern**: Business logic isolated in service modules
2. **Utility Modularity**: Shared helpers prevent code duplication (~150 lines eliminated)
3. **Type Safety**: Strong TypeScript typing with generics
4. **Separation of Concerns**: Clear boundaries between UI, logic, and data
5. **Reusability**: Generic helpers used across multiple services

**Auto-Assignment Algorithm:**

The schedule auto-assignment implements a fairness-based greedy algorithm:

```
Algorithm: Enhanced Fair Assignment
Input: Unassigned shifts, Members with constraints
Output: Optimal shift assignments

1. Calculate historical points (90-day lookback)
2. For each unassigned shift:
   a. Filter eligible members:
      - Check availability constraints (never-available times)
      - Check individual limits (max shifts per week/month)
   b. Select member with lowest total points:
      - Total = Current schedule points + Historical points
   c. Apply holiday weighting (1.5x multiplier for Swedish holidays)
   d. Assign shift and update member tracking
3. Commit all assignments atomically

Constraints:
- Availability: Time-slot based restrictions per weekday
- Limits: Min/max shifts per week/month per member
- Holidays: Swedish holidays get 1.5x point multiplier
- Fairness: Â±10% point distribution across members
```

### ADR-001: Why Firebase/Firestore over PostgreSQL?

**Decision**: Use Firebase Firestore instead of PostgreSQL

**Context**:
- Need for real-time updates (schedules, shifts, notifications)
- MVP timeline pressure (need to launch quickly)
- Small team with limited backend experience
- Tight budget constraints

**Pros**:
- âœ… Real-time sync out of the box
- âœ… Offline support built-in
- âœ… Generous free tier (freemium-friendly)
- âœ… Auto-scaling (no ops overhead)
- âœ… Firebase SDK simplifies development
- âœ… Security Rules reduce backend code

**Cons**:
- âŒ Complex queries harder (no JOINs)
- âŒ Vendor lock-in to Firebase
- âŒ NoSQL requires denormalization
- âŒ Limited aggregation capabilities

**Mitigation**:
- Denormalize data for common queries
- Use Cloud Functions for complex aggregations
- Pre-compute analytics for heavy reports

**Status**: Accepted

### ADR-002: Why Cloud Run instead of App Engine?

**Decision**: Use Cloud Run Services for API backend

**Context**:
- Need containerized deployment
- Want flexibility to migrate later
- Auto-scaling requirements
- Cost optimization (scale to zero)

**Pros**:
- âœ… Portable (Docker containers)
- âœ… Scale to zero (cost savings)
- âœ… Fast cold start (<1s)
- âœ… Support any language/framework
- âœ… Easy local development

**Cons**:
- âŒ More complex than App Engine
- âŒ Need Docker knowledge

**Status**: Accepted

### ADR-003: Why Hybrid (Cloud Run + Cloud Functions)?

**Decision**: Use both Cloud Run Services AND Cloud Functions Gen2

**Context**:
- Need stateless API for user requests
- Need event-driven processing for triggers
- Different scaling characteristics

**Rationale**:
- **Cloud Run**: Long-running requests, stateful connections, RESTful API
- **Cloud Functions**: Event-driven, short-lived, background jobs

**Status**: Accepted

## ğŸ”„ Migration Path (Future)

### If Firestore becomes limiting

**Scenario**: App grows to >100 stables with complex reporting needs

**Migration Strategy**:
1. **Dual-write period**:
   - Write to both Firestore + PostgreSQL
   - Validate data consistency

2. **Read migration**:
   - Gradually move reads to PostgreSQL
   - Keep Firestore for real-time features

3. **Final cutover**:
   - Firestore only for real-time sync
   - PostgreSQL for source of truth

**Estimated effort**: 2-3 months with 2 engineers
